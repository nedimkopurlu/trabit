**What user gains:**
- Each habit can have a notification time (HH:MM format)
- When time arrives and app is open, user gets PWA push notification
- Notifications fire roughly on-time while app is active
- On iOS PWA: notifications only trigger when app/browser is running (documented limitation)

**Output:**

## Context

**Notification state flow:**
1. User sets habit notification time via HabitForm (e.g., "06:30")
2. Time stored in Firestore as `habit.notificationTime` string
3. App loads habits via `useHabits()` hook
4. On app load + every 60s: check if current time matches any habit's notification time
5. If match: fire local Notification API alert
6. User clicks notification → PWA notification system handles interaction

**iOS PWA limitation:**
- Web Push API doesn't work (requires server APNS tokens)
- Scheduled local notification APIs don't exist on WebKit
- Best-effort: polling while app is open, no background support
- Document this limitation to users

**Tech stack:**
- No new dependencies (Notification API is native, Interval is native)
- Use existing `checkNotifications()` and `fireNotification()` patterns from research
- Time parsing: "09:00" format (HH:MM), local device timezone
- localStorage is not needed (checking happens in-memory)

## Tasks

### Task 1: Create Notification Scheduler Utility

**Files:** `lib/notification-scheduler.ts`

**Action:**
Create utility module with three functions:

**1. `getNextNotificationTime(timeStr: string): Date`**
- Input: time string "HH:MM"
- Parse hours and minutes via `split(":")` and `map(Number)`
- Create `Date` object set to today at that time
- If time already passed today, move to tomorrow
- Return `Date` object for next trigger

Example: If it's 10:00 and notification is "09:00", return tomorrow 09:00

**2. `checkNotifications(habits: Habit[]): string[]`**
- Input: array of habits from `useHabits()` hook
- Filter habits that have `notificationTime` set (non-null)
- For each habit: calculate `getNextNotificationTime(habit.notificationTime)`
- Calculate milliseconds since that trigger time: `now.getTime() - nextTrigger.getTime()`
- If difference is 0 to 60000 milliseconds (1-minute window): add habitId to results array
- Return array of habitIds that should fire now
- Rationale: 1-minute window handles polling at 60s intervals

**3. `fireNotification(habitId: string, habitName: string): Promise<void>`**
- Input: habit ID and habit name
- Check if Notification API exists: `"Notification" in window`
- Check permission: `Notification.permission === "granted"`
- If not granted, log warning and return
- Create new Notification with:
  - Title: `"${habitName}"`
  - Body: "Bildirimi görmek için buraya dokunun"
  - Badge: "/icons/icon-192.png"
  - Tag: `"habit-${habitId}"` (prevents duplicates for same habit)
  - Data: `{ habitId }`
- Use native Notification constructor, no library needed

Type definitions:
```typescript
import type { Habit } from "./habit-types";

export interface ScheduledNotification {
  habitId: string;
  habitName: string;
  notificationTime: string; // HH:MM
  nextTrigger: Date;
}
```

**Verify:**
- Module exports all three functions
- `getNextNotificationTime()` parses times correctly
- `checkNotifications()` returns empty array if no times match
- `checkNotifications()` returns habit IDs if times match
- `fireNotification()` creates Notification without error
- TypeScript compiles cleanly

**Done:**

### Task 2: Create Notification Permission Utility

**Files:** `lib/notification-permissions.ts`

**Action:**
Create lightweight permission utility with one function:

**`requestNotificationPermission(): Promise<boolean>`**
- Check if Notification API exists: `"Notification" in window`
- If not supported, log warning, return false
- Check current permission: `Notification.permission`
  - If "granted": return true (already have permission)
  - If "denied": return false (user rejected, don't ask again)
  - If "default": show permission dialog via `Notification.requestPermission()`
- Return true if permission granted, false otherwise

This function:
- Never shows dialog twice to same user (browser enforces this)
- Returns immediately if already decided
- Is safe to call multiple times

```typescript
export async function requestNotificationPermission(): Promise<boolean> {
  if (!("Notification" in window)) {
    console.warn("Notification API not supported in this browser");
    return false;
  }

  if (Notification.permission === "granted") {
    return true;
  }

  if (Notification.permission === "denied") {
    console.warn("User has denied notification permission");
    return false;
  }

  // permission === "default" — ask user
  try {
    const permission = await Notification.requestPermission();
    return permission === "granted";
  } catch (err) {
    console.error("Error requesting notification permission:", err);
    return false;
  }
}
```

**Verify:**
- Function compiles without errors
- TypeScript infers Promise<boolean> return type
- Handles all three permission states

**Done:**
- `lib/notification-permissions.ts` created
- One-liner permission request ready for app integration

**On App Load (useEffect 1):**
- Trigger once when user is authenticated and has onboarded

- No error toast needed (permission request is non-critical)

```typescript
useEffect(() => {
  if (!user || !hasOnboarded) return;
  requestNotificationPermission().catch(console.error);
}, [user, hasOnboarded]);
```

**On App Load + Periodic Check (useEffect 2):**
- Trigger when: user is authenticated, onboarded, habits array loaded
- On mount: immediately check for due notifications
- Set interval to check every 60 seconds
- For each due habitId: find habit object and call `fireNotification(habitId, habitName)`
- Cleanup interval on unmount

```typescript
useEffect(() => {
  if (!user || !hasOnboarded || !habits.length) return;

  const checkAndFire = async () => {
    const dueHabitIds = checkNotifications(habits);
    for (const habitId of dueHabitIds) {
      const habit = habits.find((h) => h.id === habitId);
      if (habit) {
        await fireNotification(habitId, habit.name);
      }
    }
  };

  // Check immediately on mount
  checkAndFire();

  // Then check every 60 seconds
  const interval = setInterval(checkAndFire, 60000);

  return () => clearInterval(interval);
}, [user, hasOnboarded, habits]);
```

## Notes

**Time parsing gotchas:**
- `"09:00".split(":")` returns `["09", "00"]`
- `map(Number)` converts to `[9, 0]`
- `setHours(9, 0, 0, 0)` sets time in device's local timezone (correct)
- Don't use `new Date("2026-04-09T09:00:00Z")` — that's UTC, not local
- Always assume local device timezone; store as string "HH:MM" only

**Notification permission browser behavior:**
- First call to `requestPermission()` shows dialog
- Browser remembers choice
- Subsequent calls return immediately with stored choice
- User can revoke in browser settings (then choice becomes "denied")
- iOS PWA follows iOS system notification settings

**Polling vs. background sync:**
- Can't use Service Worker background sync (iOS PWA doesn't support it)
- Can't use scheduled local notifications (not standardized)
- Polling is only reliable approach for iOS PWA
- 60-second interval balances battery/responsiveness (typical for PWAs)
- If notification time passes while app is closed, it won't fire (documented)

**Testing on iOS PWA:**
- Notification API permission behavior differs from web browser
- iOS PWA must be opened fresh from home screen, not Safari
- Use actual device; simulator may have inconsistent notification behavior
- Document iOS PWA limitation in-app: "Bildirimleri almak için uygulamayı açık tutunuz"

**Edge cases handled:**
- Habit without notification time: `notificationTime === null` → skipped
- Notification with 0 habits: `checkNotifications([])` returns empty array
- Permission denied: `fireNotification()` logs warning, returns
- Multiple notifications in same minute: all fire (1-minute window catches them)
- App restart at exact notification time: notification may fire twice (accepted edge case, rare)
