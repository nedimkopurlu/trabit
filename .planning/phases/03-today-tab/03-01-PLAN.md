### Task 1: Create Habit Schedule Filtering Utils
**Files:** `lib/habit-schedule.ts`

**Action:**
Create timezone-aware day-of-week filtering using native Intl API (no date-fns dependencies). Export:

1. `isHabitScheduledToday(schedule, timezone?)` — Returns true if habit's schedule matches today in given timezone
2. `getUserTimezone()` — Returns user's timezone from browser (or 'UTC' on server)
3. `getTodayInTimezone(timezone)` — Returns midnight (00:00:00) in user's timezone as Date object
4. `formatDateAsKey(date)` — Converts Date to YYYY-MM-DD string (used as Firestore document ID)
5. `getTodayKeyInTimezone(timezone)` — Returns today's date as YYYY-MM-DD string

**Implementation notes:**
- Use `Intl.DateTimeFormat` with weekday and date parts separately
- Map day names to indices: Sun=0, Mon=1, ..., Sat=6
- Weekdays check: dayIndex >= 1 && dayIndex <= 5
- Weekends check: dayIndex === 0 || dayIndex === 6
- Handle timezone parameter (default to browser's Intl timezone)
- All functions work offline (no network calls)

**Test cases to consider:**
- "daily" always returns true regardless of day
- "weekdays" returns true Mon-Fri, false Sat-Sun
- "weekends" returns true Sat-Sun, false Mon-Fri
- Timezone-aware: user in UTC-5 should get correct day even if server is UTC
- Date key format is always YYYY-MM-DD and sortable

### Task 2: Create Firestore Completion CRUD Layer
**Files:** `lib/completions-db.ts`

**Action:**
Create Firestore completion schema and CRUD operations. Completions stored as subcollection at:
`/users/{uid}/habits/{habitId}/completions/{dateStr}`

Export these functions (all require Firebase Auth context to get `uid`):

1. `markComplete(uid, habitId, dateStr, amount?, quick?)` — Set completion doc with merge (idempotent)
2. `undoCompletion(uid, habitId, dateStr)` — Delete completion document
3. `getCompletionForDate(uid, habitId, dateStr)` — Query single date (returns Promise<HabitCompletion | null>)
4. `getCompletionsInRange(uid, habitId, startDate, endDate)` — Query date range for streak calculations
5. `listenToTodaysCompletions(uid, habitId, dateStr, callback)` — Real-time listener for today's completion state (returns unsubscribe function)

**Interface definition:**
```typescript
export interface HabitCompletion {
  date: string; // YYYY-MM-DD in user's timezone
  completed: boolean; // Always true (soft-delete: remove doc instead)
  amount?: number; // For "amount" type habits, current value
  quick?: boolean; // true if marked complete via "2dk" quick action
  completedAt: Timestamp; // server timestamp when marked complete
}
```

**Implementation notes:**
- `markComplete()` should use Firestore's `setDoc(ref, data, { merge: true })` for idempotency (calling twice with same data = no issue)
- `completedAt` should use `serverTimestamp()` not client time
- Date strings are user's timezone (calculated via `getTodayInTimezone()` from habit-schedule.ts)
- `listenToTodaysCompletions` should return real-time updates as document changes

**Firestore rules update:**
Add to existing rules in firestore.rules:
```
match /users/{uid} {
  match /habits/{habitId} {
    match /completions/{dateStr} {
      allow read, write: if request.auth != null && request.auth.uid == uid;
    }
  }
}
```

**Error handling:**
- Handle permission denied gracefully (user not authenticated)
- Handle network errors in listeners (offline behavior)
- Return null for missing completions (not throw)

### Task 3: Create `useHabitCompletion` Hook
**Files:** `lib/use-habit-completion.ts`

**Action:**
Create custom React hook managing habit completion state with optimistic UI + Firestore real-time sync.

The hook should:
1. Accept `habitId` as parameter
2. Return object with:
   - `isComplete: boolean` — Is habit marked complete today?
   - `amount: number` — For amount-type habits, current amount
   - `loading: boolean` — True while Firestore operation pending
   - `error: Error | null` — Last error (if any)
   - `toggleComplete(quick?: boolean)` — Toggle completion (optimistic, syncs to Firestore)
   - `setAmount(newAmount)` — Update amount for amount-type habits
3. Set up real-time listener to today's completion doc on mount
4. Handle optimistic updates (show result immediately, sync to Firestore in background)
5. Handle cleanup (unsubscribe from listener on unmount)

**Flow diagram:**
```
User clicks "Yaptım" button
  ↓
setIsComplete(true) immediately (optimistic)
  ↓
Call markComplete() async in background
  ↓
Firestore listener fires with server update
  ↓
Confirm optimistic update was correct (or revert if failed)
```

**Implementation notes:**
- Use `useCallback` for stable function references
- Use `useEffect` to set up listener on mount (depend on habitId + uid)
- Handle case where habit type is "amount" vs "boolean"
- For boolean habits, `amount` can be 0 or 1
- For amount habits, return actual amount from completion doc
- On error, revert optimistic state and show error
- If no completion doc exists, return `isComplete: false` and `amount: 0`

**Dependencies needed:**
- `useCallback, useEffect, useState` from React
- `useAuth()` hook (from Phase 1) to get uid
- `useHabits()` hook (from Phase 2) to get habit details
- Functions from completions-db.ts
- Functions from habit-schedule.ts for today's date key

## Notes

### Timezone Handling Philosophy
This plan uses user's browser timezone (via `Intl.DateTimeFormat`) for all date calculations. This means:
- ✓ Habits correctly align to user's local day, not server UTC
- ✓ Works offline (no API calls)
- ✓ iOS Safari compatible (no moment.js issues)
- ✗ User traveling to different timezone won't re-sync habits (deliberate, prevents confusion)

Future improvement (Phase 4): Store timezone in user doc, allow manual override.

### Why Firestore Subcollection?
Completions are stored under each habit (not a separate top-level collection) because:
- ✓ Scales infinitely (1 doc per day, ~365 years of data)
- ✓ Queryable by date range (for streak calculations)
- ✓ Real-time listeners work on subcollection
- ✓ Auto-cleanup when habit is deleted (subcollection cleanup)
- ✓ Matches Firebase best practices for hierarchical data

### Optimistic Update Pattern
When user marks habit complete:
1. Update React state immediately (optimistic)
2. Call `markComplete()` in background
3. Set up real-time listener (if not already set)
4. Firestore listener will fire with server data
5. Confirm optimistic update was correct (no revert needed if data matches)

This gives instant feedback even on slow networks. If operation fails, the listener won't fire, and we show error toast.
